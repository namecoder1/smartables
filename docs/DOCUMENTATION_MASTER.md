## 1. System Overview & Core Philosophy

Smartables is a SaaS for restaurants combining Table Management, Reservations, and CRM. The architecture is designed for **high compliance** and **automated scalability**.

### The "Monolithic Compliance" Pivot

Initially, we considered a "Managed Account" model (creating a sub-account for each restaurant). However, due to high minimum spend requirements ($1k/mo) on Telnyx for Managed Accounts, we have pivoted to a **Single Account Model** ("Monolithic").

**What this means:**

1.  **Infrastructure**: We use ONE Telnyx Master Account. All phone numbers live in this single account.
2.  **Logic Separation**: The backend code enforces isolation. A restaurant cannot access/control another restaurant's number.
3.  **Compliance Separation**: Even though numbers are in one account, we create **distinct Regulatory Requirements** for each phone number based on the specific location's documents. This ensures we don't violate Italian telecom laws (e.g., using a Rome address to activate a Milan number).

- **Frontend**: Next.js (App Router), deployed on Vercel.

- **Auth**: Supabase Auth (managed via `public.profiles` trigger).
- **Telecom (VoIP)**: Telnyx (Phone Numbers, SIP Trunking, Call Control).
- **Messaging**: WhatsApp Business API (via Meta Cloud API).

Since we share one Telnyx Account, all calls hit `https://api.smartables.com/api/webhooks/telnyx`.

3.  **DB Lookup**: `SELECT * FROM locations WHERE telnyx_phone_number = '+3902123456'`.
4.  **Match Found**: Load that location's settings (opening hours, AI prompt) and handle call.
5.  **No Match**: Reject call (SIP 404).

# Telnyx Compliance & Activation Flow

## 1. Overview

Smartables uses a **"Shared Fate Protection"** model.

- **Master Account**: All numbers are purchased under one main Telnyx account.
- **Risk**: If one client commits fraud (e.g., spam calls), Telnyx bans the _entire_ account.
- **Solution**: Clients cannot upload documents directly to Telnyx. They upload to us (Supabase), and an Admin manually approves the request after verifying the documents.

## 2. Detailed Flow

### Step A: Client Submission (Frontend)

1.  **User** visits `/compliance` page.
2.  **Upload**: Uploads Identity Document (PDF/JPG) and Address Proof.
3.  **Address Detection**: System suggests Area Code based on address (Google Maps API).
4.  **Submit**: User clicks "Invia Documenti".
    - **API**: Calls `/api/telnyx/compliance/submit`.
    - **Action**:
      - Saves files to Supabase Storage (`compliance-docs/{locationId}/...`).
      - Creates DB Record in `telnyx_regulatory_requirements`.
      - **Status**: `pending_review`.
      - **Email**: Sends confirmation to User ("In Revisione") AND Admin (`bartolomei.private@gmail.com`).

### Step B: Admin Review (Internal)

1.  **Admin** receives email notification.
2.  **Dashboard**: Admin visits `/manage`.
3.  **Review**:
    - Sees "Location Name", "Organization", and "Area Code".
    - Can download/view attached documents.
4.  **Action**:
    - **Reject**: Updates status to `rejected`. User gets notified (future feature).
    - **Approve**: Triggers Server Action `approveComplianceRequest`.

### Step C: Telnyx Submission (Backend - Server Action)

_Triggered ONLY by Admin Approval at Step B._

1.  **Download**: System downloads files from Supabase Storage.
2.  **Upload to Telnyx**: Uses `telnyx.documents.create` to upload files to Telnyx.
3.  **Requirement Group**: Calls `telnyx.requirementGroups.create` linking the uploaded docs.
4.  **DB Update**:
    - Status becomes `pending` (Telnyx verification status).
    - Saves `telnyx_requirement_group_id`.

### Step D: Telnyx Verification (External)

1.  Telnyx reviews the documents (24-48h).
2.  **Webhook** (To Be Implemented): Updates DB status to `approved`.
3.  **Number Purchase**: Once `approved`, user can buy numbers in that Area Code.

# 🛡️ SECURITY PROTOCOL: Preventing "Shared Fate" (Single-Account Architecture)

**Context:**
We are operating on a **Single Monolithic Telnyx Account** to save costs.
**Risk:** "Shared Fate". If one malicious user uploads fake documents, Telnyx could ban the _entire_ Master Account, taking down all clients.
**Objective:** Implement strict **Code-Level Isolation** and a **Human Firewall** to prevent bad actors from triggering Telnyx fraud detection.

---

## 1. 🧱 Database Level: "Isolation by Design"

We cannot trust input from the Frontend regarding Regulatory IDs. We must enforce strict ownership checks.

### A. SQL Constraints (Hard Enforcement)

Ensure this constraint is active to prevent logical data corruption (e.g., assigning Rome docs to a Milan number).

```sql
-- Prevents an org from having multiple active bundles for the same area code
ALTER TABLE public.telnyx_regulatory_requirements
ADD CONSTRAINT unique_org_area_code UNIQUE (organization_id, area_code);

```

### B. The "Paranoid Query" Pattern (Backend)

When purchasing a number (`POST /buy-number`), **NEVER** trust a `requirement_group_id` sent by the client. Always look it up securely using the Auth Token.

**❌ BAD (Vulnerable):**

```typescript
// DANGEROUS: Client could inject an ID belonging to another valid org
const { regulatory_id, area_code } = req.body;
await telnyx.orders.create({ regulatory_group_id: regulatory_id, ... });

```

**✅ GOOD (Secure):**

```typescript
// 1. Extract User Org from JWT
const orgId = req.user.organization_id;

// 2. Database Lookup (The only source of truth)
const { data: regulatory } = await supabase
  .from('telnyx_regulatory_requirements')
  .select('telnyx_requirement_group_id')
  .eq('organization_id', orgId)  // Enforced ownership
  .eq('area_code', req.body.area_code)
  .eq('status', 'approved')      // MUST be admin-approved
  .single();

if (!regulatory) throw new Error("Security Violation: No approved docs for this area.");

// 3. Use the trusted ID
await telnyx.orders.create({ regulatory_group_id: regulatory.telnyx_requirement_group_id, ... });

```

---

## 2. 🚦 Workflow Level: "The Human Firewall"

**Crucial Rule:** User uploads **MUST NOT** trigger Telnyx APIs immediately.
We introduce a manual review step to catch fraud/spam _before_ it reaches Telnyx servers.

### State Machine Implementation

**Step 1: User Upload (Client Side)**

- User uploads PDF.
- **Action:** Save to Supabase Storage bucket `compliance-docs`.
- **DB Update:** Insert row into `telnyx_regulatory_requirements` with `status: 'pending_review'`.
- **API Call:** **NONE.** (Do not call Telnyx).

**Step 2: Admin Review (Internal Dashboard)**

- Admin sees "New Doc". Checks if it's a valid Utility Bill/ID.
- Admin clicks **"Approve & Submit"**.

**Step 3: Submission (Server Action)**

- _Triggered only by Admin Action._
- Backend reads file from Storage.
- Backend calls `telnyx.documents.create`.
- Backend calls `telnyx.requirementGroups.create`.
- **DB Update:** Set `status: 'approved'` and save `telnyx_requirement_group_id`.

---

## 3. 🔌 Monitor Level: "Circuit Breaker" (Webhooks)

If a user bypasses our check and Telnyx rejects the document, we must react instantly to prevent account flagging.

**Webhook Event:** `regulatory_requirement.rejected`

**Handler Logic:**

```typescript
if (event.type === "regulatory_requirement.rejected") {
  const telnyxGroupId = event.payload.resource_id;
  const reason = event.payload.errors[0]?.detail;

  // 1. Find the culprit
  const { data: record } = await supabase
    .from("telnyx_regulatory_requirements")
    .select("organization_id, id")
    .eq("telnyx_requirement_group_id", telnyxGroupId)
    .single();

  if (record) {
    // 2. Mark as rejected in DB
    await supabase
      .from("telnyx_regulatory_requirements")
      .update({
        status: "rejected",
        rejection_reason: reason,
      })
      .eq("id", record.id);

    // 3. (Optional) Auto-Flag Organization
    // If we receive multiple rejections, auto-suspend the org to protect the master account.
    await alertAdmin(
      `RISK: Org ${record.organization_id} had documents rejected by Telnyx.`,
    );
  }
}
```

# 📞 Smartables Telnyx Integration Master

**Context**: Single Account Architecture (No Managed Accounts)
**Purpose**: Guide to managing Italian VoIP Numbers, Regulatory Compliance, and Webhook Routing.

---

## 1. The Regulations (Italy)

In Italy, you cannot simply "buy" a local number (e.g., 02 for Milan) anonymously.
**Requirement**: You must provide proof of identity and proof of address _in that local area_.

**Implication for Smartables**:
We cannot just have a "Buy" button. We need a "Verify -> Approve -> Buy" flow.
Every phone number purchased MUST be linked to a specific **Requirement Group** ID on Telnyx end.

---

## 2. Detailed Implementation Flow

### Phase 1: Document Upload (Frontend)

**Page**: `/compliance`
**Component**: `DocumentUploadForm`

1.  User selects the Location (e.g., "Pizzeria Roma").
2.  User uploads 2 files:
    - **Identity Proof**: Passport or ID Card of the Legal Rep.
    - **Address Proof**: Utility Bill or "Visura Camerale" showing the address in Rome.
3.  Files are uploaded to Supabase Storage: `compliance-docs/{location_id}/{filename}`.

### Phase 2: Backend Submission (API)

**Endpoint**: `(Proposed) /api/telnyx/compliance/submit`

1.  **Create Document (Telnyx)**:
    For each file in Supabase, we send it to Telnyx.
    - `POST https://api.telnyx.com/v2/documents`
    - _Payload_: Multipart form data file.
    - _Return_: `document_id`.

2.  **Create Requirement Group**:
    We bind these documents to a specific regulatory bundle (e.g., "Italy Local Geographic Number").
    - `POST https://api.telnyx.com/v2/requirement_groups`
    - _Payload_:
      ```json
      {
        "country_code": "IT",
        "phone_number_type": "local",
        "action": "order",
        "customer_reference": "Location:{location_id}",
        "requirements": [
          { "requirement_id": "req_xyz", "document_id": "doc_123" },
          { "requirement_id": "req_abc", "document_id": "doc_456" }
        ]
      }
      ```
    - _Return_: `id` (The Requirement Group ID). Save this in `telnyx_regulatory_requirements`.

### Phase 3: Purchasing the Number

**Endpoint**: `/api/telnyx/buy`

We **never** call purchase without the requirement ID.

1.  **Search**: User searches for available numbers in "06" (Rome).
2.  **Select**: User picks `+3906123456`.
3.  **Buy Request**:
    - Backend fetches the `telnyx_requirement_group_id` associated with that location from DB.
    - **Validation**: If status is `rejected` or ID is missing, throw error "Compliance Required".
    - **Execute**:
      ```typescript
      POST /v2/number_orders
      {
         "phone_numbers": [{ "phone_number": "+3906123456" }],
         "requirements_id": "group_id_from_phase_2"  // <--- CRITICAL
      }
      ```

---

## 3. Dealing with "Managed Accounts" (Future Proofing)

_Currently, we use the Master Account._
However, if we later enable Managed Accounts (Sub-accounts for customers):

1.  **No Migration**: You typically cannot move a number from Master to Sub-account easily.
2.  **Hybrid Logic**: The code in `lib/telnyx.ts` should eventually support a `managedAccountId` parameter.
    - If passed, use that Sub-account's API Token.
    - If null, use the Master API Token (Current state).

**How to Request Managed Accounts (If needed later):**
Email `support@telnyx.com`: _"I am building a SaaS platform. Please enable Managed Accounts on my master account so I can isolate customer data/billing."_

---

## 4. Webhook Handling (`/api/webhooks/telnyx`)

**URL**: `https://api.smartables.com/api/webhooks/telnyx`
**Events to Handle**:

- `call.initiated`: Incoming call. (Trigger routing logic).
- `call.answered`: Call connected.
- `call.hangup`: Call ended. (Log duration/cost).
- `message.received`: Incoming SMS (Not for WhatsApp, but for fallback).

**Security**:
Telnyx signs requests with `Telnyx-Signature-Ed25519`.
Use the `TELNYX_PUBLIC_KEY` to verify the signature in the middleware to prevent spoofing.

**Routing Implementation**:

```typescript
// Pseudo-code for Webhook
const { to } = payload.data.payload; // "+3902..."
const location = await db.locations.find({ telnyx_phone_number: to });

if (!location) {
  return telnyxCommand.reject();
}

return telnyxCommand.answer().then(() => playWelcomeMessage(location));
```

# 💬 Smartables WhatsApp (Meta) Integration Master

**Context**: Hybrid Strategy (Aggregator now, Tech Provider later)
**Goal**: Enable restaurants to receive WhatsApp messages on their Landline (Telnyx) Number.

---

## 1. Strategy Overview

We are dealing with two potential architectural models. We are starting with **Option A** but designing the code to support **Option B**.

### Option A: Aggregator (Current)

- **Concept**: Smartables S.r.l. owns ONE WhatsApp Business Account (WABA).
- **Flow**: We add the customer's Telnyx number to _our_ WABA.
- **Pros**: Fast to implement, no OAuth flow for users. User does nothing but request it.
- **Cons**: "Shared Fate". If one client spams, our WABA reputation drops, affecting all clients. Display Name must be approved by Meta and might be tricky ("Why does Smartables have a number called 'Pizza Mario'?").

### Option B: Tech Provider / Embedded Signup (Target)

- **Concept**: Each Client has _their own_ WABA.
- **Flow**: Client clicks "Login with Facebook", accepts terms, and shares WABA access with Smartables app.
- **Pros**: Isolates risk. 100% Compliant.
- **Cons**: Complex OAuth implementation. Requires "Business Verification" for Smartables Inc.

---

## 2. Implementation Guide (Aggregator Mode)

### A. Number Verification (The "Voice" Trick)

Since we are verifying a **Landline** (Telnyx Number), we cannot receive SMS.

1.  **Telnyx Setup**: Ensure the number has a Voice connection. Forward calls to your mobile temporarily if backend isn't ready.
2.  **Meta Setup**:
    - Go to WhatsApp Manager -> Add Phone Number.
    - Enter the Telnyx Number (`+3902...`).
    - Choose Verification Method: **Phone Call**.
3.  **The Handshake**:
    - Meta triggers a call to `+3902...`.
    - Telnyx routes the call to you (or your webhook).
    - You answer and hear the 6-digit code.
    - Enter the code in Meta Dashboard.
    - _Result_: Number verified and active.

### B. System User Token

Neve use temporary 24h tokens for production.

1.  **Business Settings** -> **Users** -> **System Users**.
2.  Create "Smartables Backend" (Admin).
3.  **Generate Token**:
    - Select App: `Smartables`.
    - Permissions: `whatsapp_business_management`, `whatsapp_business_messaging`.
    - Save as `META_SYSTEM_USER_TOKEN`.

---

## 3. Database Design for Hybrid Support

To support shifting to Tech Provider later without rewriting code, we store credentials dynamically.

**Table**: `organizations` (or a specific `integrations` table) should store:

- `waba_id`: The ID of the WhatsApp Account.
- `meta_access_token`: The token to use for API calls.

**Logic**:

- **Aggregator Mode**: We populate these fields with _our_ Master WABA ID and Master System User Token for every organization.
- **Tech Provider Mode**: We populate these fields with the _User's_ WABA ID and User's Token obtained via OAuth.

**Code Example**:

```typescript
async function sendMessage(orgId, to, text) {
  const org = await db.orgs.get(orgId);
  // Works for both modes transparently
  const wabaId = org.waba_id;
  const token = org.meta_access_token;

  await metaApi.post(`/${wabaId}/messages`, { ... }, { headers: { Authorization: token }});
}
```
